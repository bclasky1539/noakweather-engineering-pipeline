Fact Table Domain Models
I have created two main fact entities:

FactCurrentWeather
    Represents the current weather at a specific location and time
    Contains references to all dimension entities (City, WeatherCondition, etc.)
    Includes computed properties and methods for weather analysis
    Has factory methods for creating from API responses

FactForecast
    Represents a complete weather forecast for a location
    Contains a list of ForecastItem entities for different time points
    Includes methods for filtering forecasts by day, time, and conditions
    Provides daily summaries and extreme weather alerts

ForecastItem
    Represents a single forecast data point for a specific time
    Similar structure to FactCurrentWeather but for a future time

Key Domain-Driven Design Features
The fact models follow DDD principles:

    1. Rich Domain Model: They encapsulate both data and behavior
    2. Computed Properties: Fields like is_daytime, forecast_count are calculated
    3. Business Logic: Methods like get_forecast_for_day(), has_precipitation_upcoming()
    4. Value-Added Methods: Both models provide helpful summaries like weather_summary and daily_summaries
    5. Factory Methods: from_api_response() and from_dict() for proper object creation
